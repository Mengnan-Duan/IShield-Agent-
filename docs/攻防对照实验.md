## 实验 2：单引擎 vs 混合引擎

### 目的

验证 Rule / Semantic / UEBA 三引擎互补融合的价值。

### 方法

通过修改检测逻辑，分别测试：

| 实验组 | 描述 | 计算方式 |
|--------|------|---------|
| **仅规则引擎** | 关闭语义和 UEBA | `is_malicious = rule_alert` |
| **仅语义引擎** | 关闭规则和 UEBA | `is_malicious = sem_alert` |
| **仅 UEBA** | 关闭规则和语义 | `is_malicious = ueba_score >= 30` |
| **规则+语义** | 关闭 UEBA | `is_malicious = rule_alert or sem_alert` |
| **全混合（推荐）** | 三引擎融合 | `combined = rule×0.35 + sem×0.45 + ueba×0.20; is_malicious = combined >= 15` |

### 预期结果框架

| 实验组 | 检出数（TP） | 漏报数（FN） | 检出率 |
|--------|------------|------------|--------|
| 仅规则引擎 | ? | ? | ?% |
| 仅语义引擎 | ? | ? | ?% |
| 仅 UEBA | ? | ? | ?% |
| 规则+语义 | ? | ? | ?% |
| **全混合** | ? | ? | **?%** |

### 互补价值分析

```
规则仅检出（rule_only）     = ? 条
语义仅检出（sem_only）      = ? 条
UEBA 仅检出（ueba_only）    = ? 条
规则+语义联合检出           = ? 条
三引擎全部检出              = ? 条

结论：全混合检出率 = max(各单引擎) + rule_only + sem_only + ueba_only + 联合检出
```

### 融合权重合理性

```
Rule × 0.35 + Semantic × 0.45 + UEBA × 0.20

语义引擎权重最高（0.45）：
  - 理由：语义分析覆盖模式外攻击（如渐进式记忆污染、纯英文绕过）
  - 但语义引擎在 API 不可用时会降级为本地模式
  - 本地模式依赖关键词组合，漏报率相对较高

规则引擎中等权重（0.35）：
  - 理由：规则对已知攻击向量（指令覆盖、SQL注入）响应快、准确率高
  - 规则命中 = 直接告警，无需等待语义推理

UEBA 权重最低（0.20）：
  - 理由：UEBA 主要用于行为异常检测，不直接参与内容检测
  - UEBA 异常时主动下调检测阈值（10/30/60），确保行为攻击不被漏过
  - UEBA 依赖 IP/Token 历史基线，单独使用时检出率有限
```

### 运行命令

```bash
cd backend
python -c "
import json
from services.rule_engine import rule_detect
from services.semantic import semantic_detect_local
from services.ueba import get_ueba_engine
from services.test_report import load_test_suite

suite = load_test_suite()
results = {'rule': [], 'sem': [], 'ueba': [], 'rule_sem': [], 'full': []}

for case in suite:
    text = case.get('attack_text', '')
    expected = case.get('expected_result', 'malicious')

    # 仅规则
    rule_alert, _, _, _ = rule_detect(text)
    # 仅语义
    sem_result = semantic_detect_local(text)
    sem_alert = sem_result[0]
    # 仅 UEBA
    ueba_engine = get_ueba_engine()
    ueba_anomaly = ueba_engine.get_context_anomaly(text, '127.0.0.1', None)
    ueba_score = ueba_anomaly.get('score', 0)

    is_mal = expected == 'malicious'

    results['rule'].append(is_mal == rule_alert)
    results['sem'].append(is_mal == sem_alert)
    results['ueba'].append(is_mal == (ueba_score >= 30))
    results['rule_sem'].append(is_mal == (rule_alert or sem_alert))
    results['full'].append(is_mal == (rule_alert or sem_alert or ueba_score >= 30))

for k, v in results.items():
    tp = sum(v)
    fn = len(v) - tp
    print(f'{k}: TP={tp} FN={fn} Rate={round(tp/len(v)*100,1)}%')
"
```

## 实验 3：NFKC 归一化防御效果

### 目的

验证 Unicode 归一化（NFKC）对混淆变体攻击的防御价值。

### 原理

```
原始攻击文本（混淆前）: "Ignor\u200Be all previous instructions"
  ↓ NFKC 归一化
归一化后文本: "Ignore all previous instructions"
  ↓ 规则检测
命中 SIG002: "Ignore all previous instructions"
```

### 测试场景

| 攻击类型 | 原始文本 | 混淆手段 | NFKC 后效果 |
|---------|---------|---------|------------|
| 同形字注入 | `adminа`（а=Cyrillic） | Cyrillic а 替代 Latin a | `adminа` → `admina`（差异仍存在，需同形字映射表） |
| 零宽字符注入 | `Ignore\u200B all` | 插入零宽空格 | `Ignore all`（NFKC 直接去除） |
| 组合攻击 | `[SYS]\u200BMSG override` | 零宽 + XML 标签 | 归一化后规则匹配 |

### 归一化防御效果评估

```bash
cd backend
python -c "
from services.test_report import load_test_suite, run_test_case
from services.rule_engine import rule_detect
from utils.normalize import normalize_text
import json

suite = load_test_suite()
# 筛选含特殊字符的样本
obfuscated = [c for c in suite if any(ord(ch) > 0x7F for ch in c.get('attack_text', ''))]
print(f'含特殊字符样本: {len(obfuscated)} 条')

# 统计归一化前后命中差异
before_hit = sum(1 for c in obfuscated if rule_detect(c['attack_text'])[0])
after_hit  = sum(1 for c in obfuscated if rule_detect(normalize_text(c['attack_text']))[0])

print(f'归一化前命中: {before_hit}')
print(f'归一化后命中: {after_hit}')
print(f'提升: +{after_hit - before_hit} 条（+{round((after_hit-before_hit)/max(len(obfuscated),1)*100,1)}%）')
"
```

### 归一化防御覆盖的变体类型

```
零宽字符去除:   \u200B \u200C \u200D \uFEFF 等控制字符
全角转半角:      Ideographic Space(U+3000) → ASCII space
同形字替换:     Cyrillic а(U+0430) → Latin a(U+0061)
                Greek  ο(U+03BF)  → Latin o(U+006F)
组合字符规范化:  组合标记字符被合并到基字符
```

### 判定标准

| 指标 | 目标 |
|------|------|
| 零宽字符类样本检出率提升 | ≥ +10% |
| 同形字类样本检出率提升 | ≥ +5% |
| 整体变体样本归一化后检出率 | ≥ 85% |
