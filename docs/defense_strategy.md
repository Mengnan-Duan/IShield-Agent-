## 1. 概述

本文档描述 IShield Agent Security Platform 的完整防御策略体系，涵盖三大核心防御维度：

| 防御维度 | 技术手段 | 防御目标 |

| 上下文隔离 | 会话级隔离、消息角色标记、记忆分层 | 阻止跨会话污染、记忆中毒 |

## 2. 上下文隔离机制（Context Isolation）

### 2.1 问题定义

在多轮对话 Agent 中，攻击者可能通过在对话历史中逐步注入恶意内容来污染 Agent 的上下文，产生以下威胁：

- **记忆中毒（Memory Poisoning）**：在多轮对话中植入伪造的系统消息或角色设定
- **会话状态伪装（Session State Forgery）**：声称自己是已验证用户或管理员
- **跨会话污染（Cross-session Pollution）**：试图从历史对话中提取其他用户的数据
- **上下文逃逸（Context Escape）**：利用对话历史绕过单轮检测

### 2.2 防御架构

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  第1层：输入解析与角色标记                            │
│  - 区分 system / user / assistant / developer 角色   │
│  - 检测伪造角色标记（如 [SYS_MSG]、[系统消息]）      │
│  - 拒绝来自 user 角色的系统级指令                     │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  第2层：上下文完整性验证                              │
│  - 验证对话历史中的指令来源是否合法                   │
│  - 拒绝 user 消息中的指令覆盖请求                    │
│  - 对话状态哈希校验（防止历史篡改）                   │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  第3层：会话级隔离存储                                │
│  - 每个会话独立存储，互不共享                         │
│  - 会话间无记忆传递（除非显式 RAG 检索）             │
│  - 长期记忆仅存储经过安全审核的内容                   │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
              模型推理引擎
```

### 2.3 关键设计

#### 2.3.1 消息角色强制标注

在 IShield 的工具调用管道中，每个用户消息都必须带有明确来源标识：

```
class MessageRole(str, Enum):
    USER      = "user"        # 用户输入（不可信）
    SYSTEM    = "system"      # 系统指令（仅管理员注入）
    ASSISTANT = "assistant"   # Agent 响应（可信）
    TOOL      = "tool"        # 工具返回（可信）
    EMBEDDED  = "embedded"    # 嵌入内容（不可信，需检测）
```

**防御规则：**
- `role=embedded` 的消息中的任何指令性内容都需要经过检测管道
- 来自 `role=user` 的消息中不得包含 `[SYS_MSG]`、`[系统消息]`、`[Session: ...]` 等标记
- 禁止在 user 消息中直接设置会话状态变量

#### 2.3.2 对话状态哈希

为防止攻击者篡改对话历史，每个会话维护一个状态哈希：

```
session_state_hash = SHA256(
    session_id +
    system_prompt_hash +
    previous_messages_hash +
    session_start_time
)
```

每次对话轮次结束后重新计算，若哈希值不匹配则触发告警并拒绝加载历史。

#### 2.3.3 会话级隔离存储

```
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│   会话 A         │  │   会话 B         │  │   会话 C         │
│  [独立上下文]    │  │  [独立上下文]    │  │  [独立上下文]    │
│  [独立记忆]      │  │  [独立记忆]      │  │  [独立记忆]      │
│  [独立状态哈希]  │  │  [独立状态哈希]  │  │  [独立状态哈希]  │
└─────────────────┘  └─────────────────┘  └─────────────────┘
         │                    │                    │
         └────────────────────┴────────────────────┘
                              │
                    ┌──────────▼──────────┐
                    │   共享知识库 (RAG)   │
                    │  [仅管理员可写入]    │
                    │  [每次检索都需审核]  │
                    └─────────────────────┘
```

### 2.4 长期记忆安全策略

长期记忆（Long-term Memory）是 Agent 保持"知识连续性"的核心，但也容易被污染：

**安全原则：**
1. **写入审查**：所有写入长期记忆的内容必须通过检测管道
2. **来源标记**：每条记忆标注来源会话和消息 ID
3. **记忆分区**：敏感信息（如用户偏好）与业务知识分区存储
4. **记忆过期**：定期清理会话级临时记忆，防止污染积累
5. **记忆验证**：检索记忆时验证来源可信度，丢弃低可信度条目

### 2.5 上下文隔离实现建议

以下伪代码描述上下文隔离的核心逻辑：

```python
def process_user_input(user_message: str, session: Session) -> ChatMessage:
    # Step 1: 解析消息角色和来源
    parsed = parse_message(user_message)

    # Step 2: 检测伪造角色标记
    if parsed.has_forged_system_tags():
        log_security_event("FORGED_SYSTEM_TAG", session.id)
        raise SecurityBlocked("检测到伪造的系统指令标记")

    # Step 3: 对话历史完整性验证
    if not session.verify_state_hash():
        log_security_event("SESSION_HASH_MISMATCH", session.id)
        raise SecurityBlocked("对话状态哈希校验失败")

    # Step 4: 检测上下文注入模式
    detection_result = hybrid_detect(parsed.content)
    if detection_result.is_malicious:
        log_security_event("CONTEXT_INJECTION", session.id, detection_result)
        raise SecurityBlocked(detection_result.reason)

    # Step 5: 角色强制标注
    safe_message = ChatMessage(
        role=MessageRole.USER,
        content=parsed.content,
        session_id=session.id,
        verified=True,
    )

    # Step 6: 更新会话哈希
    session.update_state_hash(safe_message)

    return safe_message
```

## 3. 输入过滤管道（Input Filtering）

### 3.1 多层检测架构

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  Layer 0: 格式规范化                                 │
│  - Unicode NFKC 规范化（归一化同形字）               │
│  - 控制字符过滤（\x00-\x1f 去除）                    │

                       │
                       ▼
┌─────────────────────────────────────────────────────┐

│  - SQL 注入模式检测（加权关键词组合）                │

                       ▼
┌─────────────────────────────────────────────────────┐
```

### 3.2 Unicode 归一化防御

Unicode 同形字攻击利用视觉相似性绕过字符串匹配：

| 攻击示例 | 归一化前 | 归一化后 |
|---------|---------|---------|
| `ignοre` (希腊字母 ο) | 未匹配 `ignore` | `ignore` |
| `administrator` (零宽插入) | 未匹配 | 归一化后 |
| `‮test‮` (RTL 覆盖符) | 混淆 | 移除控制符 |

**实现：**

```python
import unicodedata

def normalize_unicode(text: str) -> str:
    """NFKC 归一化 + 控制字符过滤"""
    # NFKC 归一化：将兼容字符映射到标准形式
    text = unicodedata.normalize("NFKC", text)
    # 移除零宽字符
    text = "".join(c for c in text if not _is_zero_width(c))
    return text

def _is_zero_width(c: str) -> bool:
    return unicodedata.category(c) in ("Cf", "Cc", "Cs")
```

### 3.3 输出脱敏（Output Filtering）

Agent 响应同样需要过滤，防止：

1. **敏感信息泄露**：Agent 不经意输出系统提示、API 密钥、数据库内容
2. **恶意代码泄漏**：Agent 生成的攻击脚本被直接利用
3. **个人信息泄露**：训练数据中的私人信息

**输出过滤策略：**

| 内容类型 | 检测方式 | 处理动作 |
|---------|---------|---------|
| 系统提示词 | 正则匹配 `You are a.*AI|你的系统提示` | 脱敏替换 |
| API 密钥格式 | 正则 `[a-zA-Z0-9]{32,}` | 替换为 `[REDACTED]` |
| 身份证号 | 正则 `\d{17}[\dXx]` | 替换为 `[REDACTED]` |
| 手机号 | 正则 `1[3-9]\d{9}` | 替换为 `[REDACTED]` |
| 恶意代码片段 | 规则 + 语义检测 | 警告标记 |

## 4. 模型行为监测（Model Behavior Monitoring）

### 4.1 工具调用行为图谱

每次工具调用记录形成行为图谱，用于异常检测：

```
ToolCall {
    session_id:     str    # 所属会话
    tool_name:      str    # 工具名
    params:         str    # 参数（哈希存储）
    decision:       str    # allow / block / confirm
    timestamp:      float  # Unix 时间戳
    latency_ms:     float  # 执行耗时
    model_response: str    # 模型对本次调用的响应（哈希）
}
```

### 4.2 异常检测规则

| 异常类型 | 检测指标 | 阈值 | 动作 |
|---------|---------|------|------|
| 短时高频调用 | 同一工具 10 秒内调用次数 | > 5 | 告警 |
| 敏感工具调用 | 调用 `send_email`/`query_db` | 任意 | 审计 |
| 执行时间异常 | 工具调用耗时 | > 95th percentile | 告警 |
| 拒绝率异常 | 单会话拦截率 | > 80% | 告警 |
| 模式漂移 | 会话内工具调用种类 | 新工具数 > 5 | 告警 |

### 4.3 实时告警机制

```
异常检测 ──► 告警级别判定 ──► 告警分发
                            │
                            ├──► SSE/WebSocket 前端实时推送
                            ├──► 日志持久化（SQLite）
                            └──► 样本归档（malicious_samples 表）
```

## 5. 工具调用劫持防御

### 5.1 工具调用安全检查流程

```
工具调用请求
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  Step 1: 参数格式校验                                │
│  - JSON schema 验证                                  │
│  - 危险字符过滤（; & | $ 等）                       │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  Step 2: 注入检测（hybrid_detect）                  │
│  - 检测参数中的提示注入                              │
│  - 检测参数中的指令覆盖                              │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  Step 3: 策略引擎评估（PolicyEngine）              │
│  - 工具 + 参数双重匹配                               │
│  - allow / block / confirm 三态决策                 │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
                决策执行
                ├── ALLOW  ──► 执行工具
                ├── BLOCK  ──► 记录 + 告警 + 拒绝
                └── CONFIRM ──► 暂停 + 等待确认
```

### 5.2 策略配置示例

详见 `backend/policies/default_policy.json`，核心策略包括：

- `POL-001`：危险表操作拦截（DROP/DELETE/TRUNCATE）
- `POL-002`：SQL 注入模式拦截
- `POL-006`：路径遍历攻击拦截
- `POL-007`：系统文件访问拦截（/etc/passwd 等）
- `POL-008`：API 密钥暴露拦截

## 7. 总结

| 防御层 | 技术手段 | 防护对象 |
|-------|---------|---------|
| 上下文隔离 | 角色标记、会话哈希、记忆分层 | 记忆中毒、跨会话污染 |
| 输入过滤 | 多层检测管道、Unicode 归一化 | 提示注入、Unicode 混淆 |
| 工具劫持防御 | 策略引擎、参数校验 | 恶意工具调用 |
| 行为监测 | 审计日志、异常检测、SSE 推送 | 异常行为发现 |
| 输出脱敏 | 正则 + 规则过滤 | 敏感信息泄露 |

本防御体系通过**深度防御（Defense in Depth）**原则，在每个环节均部署检测和阻断机制，确保即使某一层被绕过，仍有其他层级提供保护。
